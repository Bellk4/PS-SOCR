# GLM-OCR Server 使い方

このドキュメントは運用設定と API 利用方法に特化しています。
インストール手順は INSTALL.md を参照してください。

## 1. 主要機能

- GLM-OCR 推論: text / table / formula / extract_json
- PDF 入力: pypdfium2 でページ画像化
- 進捗表示: 事前処理中 -> OCR 中
- 実行中断: 中断 API + UI ボタン
- 改行後処理: none / paragraph / compact
- レイアウト検出: PP-DocLayoutV3（paddleocr）
- クロップ領域指定: crop_region
- OAuth ログイン保護（任意）

## 2. 環境変数設定

プロジェクト直下の .env を使用します。

例:

```env
HOST=0.0.0.0
PORT=9000
TORCH_CHANNEL=cu126
AUTH_DISABLED=1
```

主な設定:

- HOST: バインドアドレス（既定 0.0.0.0）
- PORT: 起動ポート（既定 8000）
- TORCH_CHANNEL: PyTorch チャネル（例: cu126, cpu）
- AUTH_DISABLED: 1 で認証無効、0 または未設定で認証有効
- AUTH0_DOMAIN
- AUTH0_CLIENT_ID
- AUTH0_CLIENT_SECRET
- AUTH0_SECRET
- AUTH0_REDIRECT_URI

## 3. 認証切り替え

- 認証無効: AUTH_DISABLED=1
- 認証有効: AUTH_DISABLED=0 または未設定

設定変更後はプロセス再起動（Docker は再作成）してください。

## 4. モデル保存先

モデルキャッシュは既定でプロジェクト内に保存されます。

- models/hf_cache
- models/hf_home

必要に応じて GLM_MODEL_CACHE で明示できます。

## 5. API

### GET /api/status

実行環境状態を返します。

主な返却項目:

- cuda_available
- device_default
- model
- model_cache_dir
- oauth_enabled
- oauth_base_path

### POST /api/analyze

マルチパートフォームで OCR を実行します。

フォーム項目:

- file（必須）
- device: auto / cuda / cpu
- dpi: 36-600
- task: text / table / formula / extract_json
- linebreak_mode: none / paragraph / compact
- schema: task=extract_json のとき必須
- max_new_tokens: 1-32768
- temperature
- use_layout
- layout_backend: ppdoclayoutv3 / none
- reading_order: auto / ltr_ttb / rtl_ttb / vertical_rl
- region_padding: 0-256
- max_regions: 1-1000
- region_parallelism: 1-8
- crop_region（JSON）
- request_id（省略時は自動採番）

### GET /api/progress/{request_id}

進捗状態を返します。

state の値:

- preprocessing
- ocr
- done
- error
- canceled
- cancel_requested

### POST /api/cancel/{request_id}

実行中ジョブに中断要求を送信します。

## 6. 画面アクセス

- UI: http://localhost:8000/
- API Docs: http://localhost:8000/docs

## 7. VB.NET アプリからの利用

VB.NET アプリが OCR サーバーを呼び出す際は、user_id を固定して送信することで、ユーザー単位の結果管理ができます。

### 設定ファイル例（VB.NET アプリ側）

```ini
[OCRServer]
BaseUrl=http://localhost:8000
UserId=app-vb-user-001
```

または JSON:

```json
{
  "OCRServer": {
    "BaseUrl": "http://localhost:8000",
    "UserId": "app-vb-user-001"
  }
}
```

### API 呼び出し例（VB.NET）

1. OCR 実行:

```
POST /api/analyze
Content-Type: multipart/form-data

file: <画像またはPDFファイル>
user_id: app-vb-user-001
task: text
device: auto
```

**C# 実装例:**

```csharp
using System.Net.Http;
using System.Net.Http.Json;
using System.IO;

var userId = "app-vb-user-001";  // 設定ファイルから読み込み
var filePath = @"C:\path\to\image.pdf";

using (var client = new HttpClient())
{
    using (var form = new MultipartFormDataContent())
    {
        // ファイルを読み込み
        var fileContent = new StreamContent(File.OpenRead(filePath));
        form.Add(fileContent, "file", Path.GetFileName(filePath));
        
        // パラメータを設定
        form.Add(new StringContent(userId), "user_id");
        form.Add(new StringContent("text"), "task");
        form.Add(new StringContent("auto"), "device");
        
        // リクエスト送信
        var response = await client.PostAsync("http://localhost:8000/api/analyze", form);
        var resultJson = await response.Content.ReadAsStringAsync();
        
        // request_id を結果から抽出
        // JsonDocument.Parse(resultJson).RootElement.GetProperty("request_id").GetString()
    }
}
```

2. 最新結果取得:

```
GET /api/latest_result?user_id=app-vb-user-001
```

**C# 実装例:**

```csharp
using (var client = new HttpClient())
{
    var userId = "app-vb-user-001";
    var response = await client.GetAsync($"http://localhost:8000/api/latest_result?user_id={userId}");
    
    if (response.IsSuccessStatusCode)
    {
        var resultJson = await response.Content.ReadAsStringAsync();
        // JSON をパースして結果を処理
        // JsonDocument.Parse(resultJson) で解析可能
    }
    else if (response.StatusCode == System.Net.HttpStatusCode.NotFound)
    {
        Console.WriteLine("結果が見つかりません。");
    }
}
```

3. 進捗確認:

```
GET /api/progress/{request_id}
```

ここで `request_id` は /api/analyze のレスポンスから取得します。

### 結果の永続性

- `user_id` を指定して /api/analyze を実行した結果は、メモリに保存されます。
- サーバー再起動後は失われます。
- 永続化が必要な場合は、レスポンスを VB.NET 側で保存してください。

### user_id の命名規則

user_id は任意の文字列ですが、識別しやすいように以下を推奨:
- `app-vb-user-001` など、アプリケーション＋用途を含める
- 複数の VB.NET インスタンスを運用する場合は、インスタンスごとに異なる user_id を使用
- 本運用では、端末 ID や ユーザー ID などをプレフィックスに含める

### 実装のベストプラクティス

1. **設定ファイルから user_id を読み込む**
   - ハードコードを避け、XML、INI、JSON など設定ファイルで管理
   - アプリ起動時に一度だけ読み込み

2. **エラーハンドリング**
   - 404 → 結果が無い（まだ処理中またはサーバー再起動後）
   - 400 → パラメータ値が不正
   - 500+ → サーバーエラー（ログ確認）

3. **タイムアウト設定**
   - large PDF や複雑な画像は処理時間が長くなる場合があります
   - クライアント側で充分な timeout を設定

## 8. よくある運用メモ

- extract_json では schema が必須
- .env はプロジェクトルートで一元管理
- Docker 運用で .env を変更した場合はコンテナ再作成が必要
- VB.NET 連携時は user_id を統一して送信することで、結果が正しく関連付けられます
- Auth0 ログイン済みのリクエストでは、Bearer のユーザーIDと user_id が不一致の場合 403 になります
